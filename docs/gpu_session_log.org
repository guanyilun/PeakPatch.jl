* Architectural decisions

** Two-kernel pipeline (gather + post-process) instead of fusion

Decision: =analyse_peak_gpu_cuda= launches two kernels — gather, then
post-process — rather than one fused kernel.

Rationale (recorded after exploring the trade-off with the user):
- Fused kernel fits more of the work into one launch (saves ~10µs
  launch overhead). Negligible compared to the few-ms kernel runtime.
- Fused kernel makes intermediate values (rad, Fbar, m0, RTHL) impossible
  to inspect when debugging. Two-kernel preserves the gather output as a
  global-memory snapshot.
- Phases 4–7 of analyse_peak run sequentially on thread 0; fusing only
  helps if other threads can do useful work concurrently. They can't (the
  algorithm is inherently sequential per peak in those phases).

The 2026-04-14 session validated this choice empirically: post-process
kernel takes ~2 ms / 100 peaks at N=512, dwarfed by the gather (~400 ms).
Fusion would save <1% wall clock.

** Float64 transfer arithmetic in =isolated_convolve_gpu=

Inside =_transfer_kernel!=, frequencies (kx, ky, kz) and the transfer
function multiplications use Float64 even though the rfft buffer is
ComplexF32. This matches the CPU =_isolated_convolve= which casts
=Float64(kx_arr[ix])= and computes =amp * kernel_fn(kx, ky, kz, k2)=
in Float64 before truncating back to ComplexF32 on store.

Without Float64 in the kernel, n=64+ shows ~1e-5 relerr on 1LPT/2LPT and
~1e-2 on phi_ij off-diagonal terms. With Float64, all variants reach
~7e-7 (Float32 round-off floor).

Cost: Float64 ops are 1/32 throughput of Float32 on consumer Ampere, but
the transfer kernel is memory-bound (one rfft cell read, one write per
thread), not compute-bound. Float64 ops here add no measurable latency.

** Hermitian-plane symmetrisation (cuFFT vs FFTW workaround)

cuFFT's =CUFFT_C2R= handles non-Hermitian-symmetric inputs at the DC
(ix=1) and Nyquist (ix=n2/2+1) planes differently from FFTW. When a
transfer function violates the Hermitian-conjugate-pair constraint at
these planes (e.g., =-kx*ky/k²= for phi_xy where ky changes sign in
the conjugate pair while kx stays the same), the GPU output diverges
from CPU by 5% at n=64.

Fix: =_hermitize_planes_kernel!= averages each conjugate pair on the
DC and Nyquist planes after the transfer, before irfft. Implementation
detail: the lower-indexed cell of each pair does the work and writes to
both slots, avoiding read/write races within the kernel.

This was the trickiest debug of the session. Discovery path:
1. Initial test at n=32 passed with all 5 transfer variants.
2. n=64 failed for phi_xy with relerr 5%, 1LPT_y with 1e-5.
3. First hypothesis: Float32 log() precision in P(k) lookup. Wrong —
   moving to Float64 didn't help phi_xy.
4. Second hypothesis: kernel arithmetic precision. Promoting to Float64
   throughout fixed 1LPT/2LPT (down to 7e-7) but phi_xy stayed at 5%.
5. Decisive test: bypass GPU kernel entirely and apply CPU transfer to
   cuFFT-rfft output, then irfft on GPU. STILL 5% relerr. So the issue
   is not in the kernel — it's in cuFFT.irfft's handling of the data.
6. Next test: cuFFT.rfft + apply CPU transfer + FFTW.irfft. relerr 3e-7
   ✓. Confirmed cuFFT.irfft is responsible.
7. Symmetrise the DC + Nyquist planes (averaging conjugate pairs) →
   matches CPU at all sizes.

* Bug log

** Race condition in Hermitian symmetrisation kernel

First implementation had each thread read its conjugate-pair neighbour
and write its own slot. With concurrent reads and writes within the
same kernel, threads were getting inconsistent (sometimes already-modified)
values from their pairs.

Symptom: averaging didn't fully fix phi_xy (left ~1e-3 relerr).
Fix: only the lower-indexed thread of each pair does the work,
writing to both slots. Race eliminated.

** Off-by-one in =_isolated_convolve_gpu= P(k) table indexing — n/a

Nothing to record here; the lookup correctly clamps below table bounds.

* Empirical findings

** GPU memory bandwidth (PCIe upload)

Measured 5–13 GB/s for host→GPU transfer of Float32 fields with
non-pinned host memory. PCIe 4.0 x16 theoretical peak is ~32 GB/s.
Half the theoretical performance is consistent with non-pinned (page-
faulted) transfers.

Workaround if it matters later: =CUDA.pin(host_array)= + async copies.
Not pursued because the bottleneck moves once Phase 1 (cuFFT) keeps
fields on the GPU between FFT and shell analysis.

** Shell-analysis crossover (N=512 grid, varying peak density)

| npeaks | CPU-1T | CPU-64T | GPU total | GPU kernel |
|--------+--------+---------+-----------+------------|
|    522 |  3.17s |   0.30s |     0.87s |      0.41s |
|   2084 |  4.39s |   0.49s |     0.82s |      0.41s |
|   5090 |  7.47s |   0.83s |     0.95s |      0.62s |
|   9861 | 12.80s |   1.39s |     1.57s |      0.74s |
|  18494 | 22.70s |   1.36s |     0.79s |      0.38s |

- GPU kernel beats CPU-64T at ~2k peaks (1.2× → 3.6× by 18k).
- GPU end-to-end (incl. ~500 ms field upload) beats CPU-64T at ~15-20k
  peaks (1.7× at 18k).
- CPU 64-thread scales sub-linearly (8-9× over 1T), Amdahl/bandwidth
  limited.
- GPU kernel scaling is sub-linear: 36× more peaks (522→18494) takes
  *less* wall time (~0.4s constant) because per-peak overhead dominates
  at small batches and amortises beautifully at large batches.

Wired =isolated_convolve_gpu= into =run_multitile_split= via a new
=use_gpu::Bool=false= keyword. On =use_gpu=true= the per-tile isolated
FFTs (1× δ + 3× ψ1 + 3× ψ2 + 1× Laplacian = 8 per tile) go through

Validation (=test/test_multiresolution_gpu.jl=):
- =ntile=2=, =n=68=, 1LPT: CPU=2127 halos / GPU=2127 halos; top-10
  halos Δpos = (0,0,0), same RTHL → bit-identical.
- Same for 2LPT.

Benchmark (=bench/bench_multitile_gpu.jl=, ntile=2, nmesh=192, 32
threads, RTX A5000, ilpt=2):
| Variant | Wall | Halos |
|---------+------+-------|
| CPU     | 213.05 s | 20467 |
| GPU     | 113.60 s | 20467 |
| Speedup | 1.88×    | 0 Δ   |

Why only 1.88× when the isolated FFT itself is 17-20× faster:
- 8 tiles × 8 isolated FFTs = 64 FFTs, but these are interleaved with
  shell analysis (CPU), tile-local 2LPT FFTs (CPU), peak finding (CPU),
  coarse-grid setup (CPU). Only the isolated FFTs were ported.
- Amdahl bound given shell-analysis + tile-local work is still on CPU.

Follow-up upside available if needed: swap shell analysis to the
already-validated =analyse_peak_gpu_cuda= (currently on the CPU here so
=zvir_half= and the formation-redshift sweep are preserved). Requires
adding =zvir_half= to the GPU post-process output first.

* Lessons learned

** Validate against production code where possible
The strongest tests are device primitives (eig3, trilinear,
kernel_strain) compared against the production CPU implementations
(LAPACK, Interpolations.jl, =RadialShell.kernel_strain=). These caught
no bugs but provide the confidence that the math is right.

For gather kernels, hand-coded CPU references in test_shell_gpu.jl
were used because the production code's interface (per-peak loop with
shared mask) doesn't expose the per-peak per-shell sums in the form
the gather kernel produces. This is a slightly weaker check but still
useful: if my GPU kernel had a bug AND my CPU reference reproduced the
same bug AND the production code was different from both, only the
end-to-end =analyse_peak_gpu_cuda= test would catch it. That test does
exist and uses the production CPU code directly.

** GPU library quirks aren't documented
The cuFFT vs FFTW Nyquist-plane divergence is mentioned nowhere in
either library's docs. Discovery required isolating the irfft step
specifically. Recommend keeping a "library quirks" subsection in any
porting effort like this.

** Memory upload often dominates GPU wall-clock for batch workloads
At the shell-analysis scale we tested, ~60-80% of GPU wall time was
in PCIe upload of input fields (4 GB at N=512). This is invisible if
fields are generated on the GPU upstream (which is the goal once
Phase 1 cuFFT replaces CPU FFTW in =run_multitile_split=).

** "Just measure it" beats theoretical analysis
Round-trip elimination (predicted to save tens of ms) was within
benchmark noise (a few % of total wall time). Upload bandwidth was the
real bottleneck. Expensive optimisations should be measured first.
