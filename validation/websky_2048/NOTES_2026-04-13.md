### Memory estimate (current implementation)

### Memory optimization: `run_multitile_lowmem` (implemented)

The standard approach stores all displacement fields on the full N³ grid
simultaneously. The lowmem approach computes each field one at a time,
extracts tiles or uses it on the fly, then frees the full-grid array.

**Serial lowmem (`run_multitile_lowmem`):**
Keeps only `delta_k_full` and `src2_k` in memory. During shell analysis,
computes each displacement field per-tile: IRFFT full grid → extract tile → free.

| Approach | Peak memory (1760³ Float64) | Extra FFTs |
|----------|---------------------------|------------|
| Standard `run_multitile` | ~500 GB (9 full-grid arrays) | 0 |
| `run_multitile_lowmem` | ~140 GB (delta_k + src2_k + temp) | ntiles×8 |

Benchmarked at N=240 (4 threads):
- Standard: 626 MB peak, 29s
- Lowmem: 364 MB peak, 41s (1.7× less memory, 1.4× slower)
- Memory ratio improves at larger N as full-grid arrays dominate overhead

**MPI lowmem (`run_multitile_mpi`, incremental src2):**
The 2LPT source computation is the bottleneck: it requires 6 phi_ij fields
in real space. The old MPI code held all 6 simultaneously. The new code
computes them incrementally, holding at most 2 at once.

For 6144³ Float32 with P MPI ranks, memory per rank:

| Component | Old MPI | New MPI (incremental) |
|-----------|---------|----------------------|
| delta_k (distributed) | 928/P GB | 928/P GB |
| 6 phi_ij (during 2LPT) | 6 × 928/P GB | 2 × 928/P GB |
| src2 accumulator | 928/P GB | 928/P GB |
| Total during 2LPT | 8 × 928/P GB | 4 × 928/P GB |

| Nodes (1 TB each) | Old per-node | New per-node | Fits? |
|-------------------|-------------|-------------|-------|
| 4 starq | 1,856 GB | **928 GB** | Yes (new only) |
| 8 starq | 928 GB | **464 GB** | Yes (both, new comfortable) |
| 16 starq | 464 GB | **232 GB** | Yes |

The incremental approach halves the 2LPT memory, making 4-node runs feasible
where the old approach needed 8+.

**Full 6144³ Websky resource estimate:**

Per octant (one independent simulation):
- Grid: 6144³, ~20 filters, ntile=8 (512 tiles)

- Time: ~24-48 hours per octant (estimated from 1760³ scaling)
- Full sky (8 octants): ~200-400 node-hours total

**Why Threefry is essential:**
The Fortran LCG cannot regenerate noise without running the full sequential
chain from scratch. Threefry's counter-based design makes regeneration O(1)
per cell: `noise[i,j,k] = threefry(seed, linear_index)`. This is the same
approach used by JAX's random number system (also built on Threefry).
The lowmem serial approach exploits this to recompute fields per tile without
storing the noise or delta_k for each displacement separately.

## Issues Encountered

1. **Filter file format**: `read_filterbank` expects first line to be the filter
   count (integer), no comment lines. Initial filter file had `#` comment headers,
   causing a parse error. Fixed by generating in the correct Fortran-compatible format.

2. **Threefry noise fill was serial**: `fill_noise_threefry!` used a plain
   `for` loop despite Threefry being counter-based (each cell independently
   computable). At 1760³ = 5.45 billion cells, this took ~20+ minutes on a
   single core. Fixed by adding `Threads.@threads` on the outer k-loop — same
   pattern as the k-space loop threading fix. With 128 threads, noise fill
   should drop to under a minute. Also fixed `fill_noise_threefry_region!`
   (the MPI variant). Fix will take effect on future runs.

3. **HDF5 extension not loaded**: `bin/peakpatch.jl` didn't `using HDF5` when
   `format="both"` was requested. The HDF5Ext package extension requires HDF5 to
   be loaded to define `write_catalog_hdf5`. Fixed by adding `@eval using HDF5`
   before the HDF5 write block.

## Validation Plan

This is a production-like run with no Fortran reference, so validation relies on
internal consistency checks and comparison against theoretical predictions.

### Sanity checks (from output log)
- [ ] **Halo count**: expect ~10-50M for this volume. Far outside → bug.
- [ ] **Per-filter peak counts**: should increase monotonically with smaller Rf.
- [ ] **Per-tile halo counts**: should be roughly uniform across 64 tiles (no
      tile boundary artifacts or missing tiles).
- [ ] **Lightcone tile pruning**: some tiles should be pruned if beyond z_max.

### Catalog analysis (`analyze_catalog.jl`)
- [ ] **Mass range**: M ~ 10¹² to 10¹⁵ Msun/h. Minimum mass set by smallest
      filter Rf=2.08 Mpc/h → M_min ≈ (4π/3) ρ_m (2.08)³ ≈ 10¹² Msun/h.
- [ ] **Redshift distribution**: halos should span z=0 to z~1.2. Count per
      z-bin should increase with z (shell volume grows as r²).
- [ ] **Position distribution**: halos fill the octant, positions span
      [-1043, +1043] Mpc/h in each dimension, no gaps at tile boundaries.
- [ ] **Number density**: n ~ 10⁻³ (Mpc/h)⁻³ for M > 10¹² is the ballpark
      from Tinker HMF at z~0.5.

### Mass function comparison
- [ ] **Raw PeakPatch HMF vs Tinker**: bin halos by mass and redshift, compute
      dn/dlnM. PeakPatch top-hat masses should be within factor ~2 of Tinker.
      Systematic offset is expected (PeakPatch uses its own collapse criterion).
- [ ] **After abundance matching**: apply `build_abundance_table` + `abundance_match`.
      The remapped N(>M) should match Tinker at each z-bin by construction.
      This is the primary quantitative validation.

### Lightcone-specific checks
- [ ] **Per-peak fcrit**: in lightcone mode, fcrit = fsc_of_z(z_peak) varies
      per peak. At z=0, fcrit ≈ 1.686. At z=1, fcrit should be higher
      (stronger collapse threshold at earlier times). Verify a sample of halos.
- [ ] **Growth factor scaling**: halo velocities should scale with D(z) × f(z).
      Check that velocity dispersion decreases at higher z.
- [ ] **Observer geometry**: halos at (x,y,z) close to observer (-1043,-1043,-1043)
      should have z≈0; halos at the far corner (+1043,+1043,+1043) should have
      z≈1.2.

### Post-processing pipeline
- [ ] **Merger**: Lagrangian exclusion should remove ~5-15% of halos (typical
      for PeakPatch).
- [ ] **Abundance matching**: build table, apply, verify mass function matches.
- [ ] **HDF5 output**: file should contain ra, dec, z, M200m columns compatible
      with XGPaint.jl `read_halo_catalog_hdf5()`.
